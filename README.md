## ЁЯЖХ Supporting a New Satisfactory Version

Game data is fully isolated per version in `01_data_vX_Y.gs` files.
Multiple versions can coexist тАФ only `GAME_VERSION` in `00_core_config.gs` is active.

```bash
# 1. Copy the template
cp src/01_data_TEMPLATE.gs src/01_data_v2_0.gs

# 2. Edit SAT.DATA['2.0'] тАФ fill MACHINES, RESOURCES, RECIPES arrays
#    Source: https://satisfactory.wiki.gg/wiki/

# 3. Activate the new version
#    In src/00_core_config.gs:  GAME_VERSION: '2.0'

# 4. Deploy
make push
```

## ЁЯПЧя╕П Architecture

### Layer 0 тАФ Config & Game Data
- `00_core_config.gs` тЖТ `SAT.CFG` тАФ configuration, utilities (`SAT.U`, `SAT.S`, `SAT.Log`, `SAT.loadGameData`)
- `01_data_v1_1.gs` тЖТ `SAT.DATA['1.1']` тАФ 18 machines, 95 resources, 67 recipes (official FR names from wiki.gg)
- `01_data_TEMPLATE.gs` тЖТ template to copy for new game versions

### Layer 1 тАФ Engine
- `10_engine.gs` тЖТ `SAT.Engine` тАФ rate calculation, flag detection, recipe index
  - Belt throughput flags: ЁЯЯа Mk.6 required (>780/min), ЁЯЯе split required (>1200/min Mk.6 limit)
- `11_solver.gs` тЖТ `SAT.Solver` тАФ objective-based production planner

### Layer 2 тАФ UI
- `20_ui_charts.gs` тАФ dashboard charts
- `30_recalc.gs` тАФ `SAT_recalcAll()` entry point

### Layer 3 тАФ Application
- `40_install.gs` тАФ sheet creation, dropdown validations, soft update (data-preserving)
- `41_triggers.gs` тАФ `onOpen()`, `onEdit()`
- `42_menu.gs` тАФ SAT menu, opens assistant by default on load

### Layer 4 тАФ Intelligence
- `50_assistant.gs` тЖТ `SAT.Assistant` тАФ full factory analysis, actionable recommendations:
  - Configuration errors, bottlenecks, high OC warnings
  - **Belt/conveyor analysis**: ЁЯЯа lines needing Mk.6 (>780/min), ЁЯЯе lines exceeding Mk.6 requiring parallel split (>1200/min)
  - Phase progression coaching, nuclear waste reminder, power budget, large surpluses
- `51_import.gs` + `51_import_ui.html` тАФ import `.sav` directly from sidebar (browser-side parsing via CDN)

## ЁЯФД CI/CD

Two workflows run automatically on every push to `main`:

| Workflow | File | Trigger | Steps |
|----------|------|---------|-------|
| **CI** | `ci.yml` | push + PR | pre-commit hooks, Jest tests |
| **Deploy** | `deploy.yml` | push to `main` | GitVersion тЖТ inject version тЖТ `clasp push` тЖТ GitHub Release |

### Version injection into GAS

The `VERSION` field in `src/00_core_config.gs` is **automatically injected** by the Deploy workflow using the GitVersion-computed semver. The deployed Apps Script code always contains the exact version that triggered the push.

### Pre-commit hooks

Hooks are defined in `.pre-commit-config.yaml`:

| Hook | Source | Purpose |
|------|--------|---------|
| `check-json`, `check-yaml` | pre-commit/pre-commit-hooks | Syntax validation |
| `end-of-file-fixer`, `trailing-whitespace` | pre-commit/pre-commit-hooks | File hygiene |
| `console-log/debug/table-detection` | chrysa/pre-commit-tools | Catch debug output in `.js` and `.gs` |
| `env-file-check` | chrysa/pre-commit-tools | Detect secrets in `.env` files |
| `check-gs-syntax` | local | Node.js syntax check on GAS files |

## ЁЯФД Versioning

Versions are computed automatically using **GitVersion** (semantic versioning from git history):

- `feat:` commits тЖТ minor bump (`3.5.x` тЖТ `3.6.0`)
- `fix:` / `chore:` commits тЖТ patch bump (`3.5.5` тЖТ `3.5.6`)
- `feat!:` or `BREAKING CHANGE` тЖТ major bump

The `CHANGELOG.md` is auto-generated by **git-cliff** on every push to `main`.
